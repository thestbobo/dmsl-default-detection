-------------------------------------------------------------------------------
1. WHAT THE ASSIGNMENT IS (in plain words)
-------------------------------------------------------------------------------
You are given data about 30,000 credit-card clients (their credit limit, some
demographics, and 6 months of bill/payment history). For each client you must
predict ONE yes/no thing:

    "Will this client DEFAULT on next month's payment?"   0 = no, 1 = yes.

You train on `development.csv` (which has the answer column) and then produce
predictions for `evaluation.csv` (which does NOT). Those predictions are scored
on a leaderboard.

WHAT "GOOD" LOOKS LIKE — and why accuracy is a trap
---------------------------------------------------
The classes are imbalanced: about 78% of clients do NOT default. So a lazy model
that always says "no default" is right ~78% of the time:

    Accuracy of "always predict 0"  = 0.78     <-- looks great, is useless
    Macro-F1 of "always predict 0"  ~ 0.44     <-- exposes that it never finds defaulters

Macro-F1 averages the F1 score of BOTH classes equally, so you only score well if
you also catch the minority "default" class. That is why this project is graded on
**macro-F1**, and why we (a) report per-class precision/recall and (b) tune the
decision threshold to maximise macro-F1 instead of blindly using 0.5.

2. WHAT EVERY FILE / FOLDER IS
-------------------------------------------------------------------------------
main.py
    The one entry point. Loads data, builds the pipeline, trains it FROM SCRATCH,
    predicts the evaluation set, and writes submission.csv. Must run in <=150 s.

src/config.py        Settings everyone shares: the random SEED (42), file paths,
                     which columns are categorical vs numeric, and the helpers
                     that robustly find the label/id columns.
src/data.py          Reads the CSVs; splits features/target; pulls the eval ids.
src/preprocessing.py Cleans the data and builds the transform (impute, scale,
                     one-hot). This is where the undocumented codes are folded.
src/models.py        Defines the candidate models AND the single "chosen" model
                     that main.py uses. Edit tuned params here.
src/evaluate.py      Cross-validation + metrics + confusion-matrix figures. This
                     produces the numbers you put in the report.
src/submission.py    Writes and validates the strict "Id,Predicted" CSV.

experiments/tune_baseline.py
                     Compares the 3 baselines and runs a small hyperparameter
                     search. Tuning lives HERE, never in main.py (exam rule).
notebooks/01_eda.ipynb
                     Exploratory data analysis: class balance, distributions,
                     the weird codes, missingness.
data/raw/            Where the dataset goes (you download it). Gitignored.
outputs/submissions/ submission.csv lands here. Gitignored.
outputs/figures/     Saved plots (confusion matrices). Gitignored.
report/              Placeholder + rules for the required IEEE report.
CLAUDE.md            Short, structured context (for LLM tools / quick reference).
requirements.txt     The Python dependencies (already installed in .venv).

4. UNDERSTANDING THE FEATURES (with examples)
-------------------------------------------------------------------------------
LIMIT_BAL   Credit limit in NT$.  e.g. 200000 = a NT$200,000 limit.
SEX         1 = male, 2 = female.
EDUCATION   1 = grad school, 2 = university, 3 = high school, 4 = other.
            (Real data also has 0, 5, 6 -> we fold these into 4 = "other".)
MARRIAGE    1 = married, 2 = single, 3 = other.
            (Real data also has 0 -> we fold it into 3 = "other".)
AGE         Age in years.

PAY_0 .. PAY_6   Repayment status, most-recent (Sept) to oldest (April).
            PAY_0 = the September status, PAY_2 = August, ... PAY_6 = April.
            -1 = paid on time (duly).
             0 = (undocumented; revolving credit / minimum payment in practice).
             1 = 1 month late,  2 = 2 months late,  ...  9 = 9+ months late.
            EXAMPLE: PAY_0 = 2  ->  the client was 2 months late on the September
            payment. PAY_0 = -1 -> paid September on time. These columns carry the
            strongest signal for default.

BILL_AMT1 .. BILL_AMT6   Bill statement amount each month (Sept .. April), NT$.
            EXAMPLE: BILL_AMT1 = 50000 -> the September bill was NT$50,000.
PAY_AMT1 .. PAY_AMT6     Amount the client actually PAID each month, NT$.
            EXAMPLE: PAY_AMT1 = 2000 with BILL_AMT1 = 50000 -> they paid only
            NT$2,000 against a NT$50,000 bill — a warning sign.

5. HOW TO ITERATE (improve the score)
-------------------------------------------------------------------------------
- Add / change a candidate model:
      edit src/models.py -> get_candidate_models()  (for comparison)
- Tune hyperparameters:
      edit/run experiments/tune_baseline.py. It prints the best params.
      Paste them into src/models.py -> CHOSEN_HGB_PARAMS so main.py uses them.
      (NEVER move tuning into main.py — it must stay fast and deterministic.)
- New preprocessing idea (e.g. engineered features):
      add it in src/preprocessing.py, inside the Pipeline, so it stays leakage-free.
- Get numbers/figures for the report:
      python experiments/tune_baseline.py
      -> prints per-model macro-F1 + saves outputs/figures/cm_*.png
      Copy these into report Sections 3 (Results) and 4 (Discussion).
